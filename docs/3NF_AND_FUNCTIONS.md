# Обоснование 3НФ и функций приложения

База данных приведена к третьей нормальной форме:

1. Основная сущность `appeals` хранит только данные обращения и внешние ключи.
2. Типы обращений вынесены в `appeal_types`.
3. Статусы вынесены в `appeal_statuses`.
4. Приоритеты вынесены в `priorities`.
5. Роли пользователей вынесены в `roles`.
6. Комментарии отделены от обращений в `appeal_comments`, потому что у одного обращения может быть несколько комментариев.
7. Скрытые внутренние метки вынесены в `appeal_flags`, потому что у одного обращения может быть несколько меток.
8. История изменения статусов вынесена в `appeal_status_history`.
9. Логи операций вынесены в `audit_logs` и локальный файл приложения.
10. Очередь синхронизации вынесена в `sync_queue` и локальный `sync_queue_local.json`.

Такой подход убирает дублирование названий статусов, типов, ролей и приоритетов. В основной таблице не хранятся повторяющиеся текстовые значения справочников, поэтому изменение названия статуса или типа не требует редактирования всех обращений.

## LWW

Для конфликтов используется стратегия Last-Write-Wins. В таблице `appeals` есть поля:

- `updated_at` — дата последнего изменения;
- `version` — версия записи;
- `last_synced_at` — дата последней синхронизации.

На стороне БД добавлена функция `upsert_appeal_lww`, а в приложении при ошибке сети изменение сохраняется в локальную очередь и повторяется при следующем обновлении.

Реализованы функции: ввод, редактирование, поиск, фильтрация, удаление, комментарии, изменение статуса, экспорт Word/Excel, ролевой доступ, логирование, синхронизация с Supabase и мобильный доступ.
